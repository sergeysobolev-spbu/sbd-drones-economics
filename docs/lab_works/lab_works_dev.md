## Описание распределения по командам разработки и хода выполнения работ

1. Оптимальный состав команды 3-4 человека
2. Роли в команде
    - лидер команды (выполняет также функции проектного менеджера)
    - архитектор
    - разработчик
    - тестировщик (QA)
    - технический писатель

    Примечание: роли можно совмещать

3. Основные этапы работы
   1. Инициация: 
        * формирование команды, закрепление ролей
        * определение формы и расписания коммуникаций (например, создание телеграм-группы для команды)
        * определение инструмента планирования (например, простой список задач в текстовом документе в репозитории или онлайн таблица)        
        * подготовка инфраструктуры для разработки для каждого участника команды (инструменты разработки, развёртывания и тестирования), 
   2. Общее планирование
        * порождение первого варианта календарного графика работ (от описания конечного результата в обратную сторону), приблизительное распределение работ по итерациям
        * продолжительность каждой итерации (спринта) 2 недели
        * детальное планирование первой итерации, согласование в команде задач, сроков и ответственных (допускается совместная работа над отдельными задачами)        
   3. Работа в итерации
        * выполнение запланированных задач
        * подготовка отчёта по итерации по шаблону:
            * краткая информация о проекте команды
            * что было запланировано (цели) на спринт [лидер команды]
            * как решили делать [архитектор], для архитектурных решений подготовить диаграммы C4
            * что сделано [архитектор/разработчик]
            * что не сделано, почему [лидер команды]
            * с каким трудностями столкнулись (техническими, организационными), если применимо
            * какие улучшение команда решила внедрить (не менее одного за спринт!) [лидер команды]
            * демонстрация нового функционала (если применимо) [архитектор/разработчик] 
            * демонстрация результатов тестирования [QA]
            * отчёт по выполнению общих пунктов ТЗ [QA]
        * оценка проектных рисков [лидер команды]
        * план на следующую итерацию
            * технические задачи
            * организационные изменения (если применимо)
            * новые гипотезы и их тестирование (если применимо), работа с рисками
            * оценка "бюджета" команды на спринт в story points (условных единицах трудоёмкости) с учётом доступности и производительности
        * формат отчёта - текстовый документ (markdown) или презентация (pptx)

4. Виды задач для команд

    1. Разработка нового компонента / сервиса

    2. Общесистемный контроль качества

        1. Разработка общесистемных сквозных (e2e) тестов

        2. Разработка общесистемных интеграционных тестов

        3. Разработка общесистемных тестов на безопасность

        4. Фоновые проверки всех подсистем

    3. Отслеживание общего хода разработки, актуализация проектных рисков

    4. Доработка и улучшение общесистемной инфраструктуры, инструментов, шаблонов документации, процессов разработки и тестирования

    5. Разработка и актуализация общесистемной документации

    6. Разработка концепции геймификации процесса

    7. Визуализация хода работы над развитием системы
