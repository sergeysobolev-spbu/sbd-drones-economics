# Описание распределения по командам разработки и хода выполнения работ

1. Оптимальный состав команды 3-5 человек
2. Роли в команде 
    - разработчиков и системных интеграторов
        - лидер команды (выполняет также функции проектного менеджера)
        - архитектор
        - разработчик
        - тестировщик (QA)
        - технический писатель
    - координаторов
        - лидер команды
        - аналитик
        - разработчик скриптов
    - тестировщиков
        - лидер команды
        - разработчик скриптов
        - тестировщик

    Примечание: роли можно совмещать

3. Основные этапы работы
   1. Инициация: 
        * формирование команды, закрепление ролей
        * определение формы и расписания коммуникаций (например, создание телеграм-группы для команды)
        * определение инструмента планирования (например, простой список задач в текстовом документе в репозитории или онлайн таблица)        
        * подготовка инфраструктуры для разработки для каждого участника команды (инструменты разработки, развёртывания и тестирования), 
   2. Общее планирование
        * порождение первого варианта календарного графика работ (от описания конечного результата в обратную сторону), приблизительное распределение работ по итерациям
        * продолжительность каждой итерации (спринта) 2 недели
        * детальное планирование первой итерации, согласование в команде задач, сроков и ответственных (допускается совместная работа над отдельными задачами)        
   3. Работа в итерации
        * выполнение запланированных задач
        * подготовка отчёта по итерации по шаблону:
            * краткая информация о проекте команды
            * что было запланировано (цели) на спринт [лидер команды]
            * как решили делать [архитектор], для архитектурных решений подготовить диаграммы C4
            * особый акцент на доверенной вычислительной базе - что в неё входит, что было сделано для её минимизации, упрощения, изоляции, контроля качества
            * что сделано [архитектор/разработчик]
            * что не сделано, почему [лидер команды]
            * с каким трудностями столкнулись (техническими, организационными), если применимо
            * какие улучшение команда решила внедрить (не менее одного за спринт!) [лидер команды]
            * демонстрация нового функционала (если применимо) [архитектор/разработчик] 
            * демонстрация результатов тестирования [QA]
            * отчёт по выполнению общих пунктов ТЗ [QA]
        * оценка проектных рисков [лидер команды]
        * план на следующую итерацию
            * технические задачи
            * организационные изменения (если применимо)
            * новые гипотезы и их тестирование (если применимо), работа с рисками
            * оценка "бюджета" команды на спринт в story points (условных единицах трудоёмкости) с учётом доступности и производительности
        * формат отчёта - текстовый документ (markdown) или презентация (pptx)

4. Виды задач для команд 

    1. Разработка нового компонента / сервиса / подсистемы    

    2. Общесистемный контроль качества

        1. Разработка общесистемных сквозных (e2e) тестов

        2. Разработка общесистемных интеграционных тестов

        3. Разработка общесистемных тестов на безопасность

        4. Фоновые проверки всех подсистем

    3. Отслеживание общего хода разработки, актуализация проектных рисков

    4. Доработка и улучшение общесистемной инфраструктуры, инструментов, шаблонов документации, процессов разработки и тестирования

    5. Разработка и актуализация общесистемной документации

    6. Разработка концепции геймификации процесса

    7. Визуализация хода работы над развитием системы

    8. Доработка каркаса нового сервиса

    9. Разработка нового "продукта", интегрирующего несколько компонентов (т.е. сборка нового "дрона" из подсистем)

    10. Доработка технико-экономической модели взаимодействия участников сети / создание новой модели

    11. Создание сквозных демонстраторов в формате jupyter блокнота

## Задачи для команды координаторов

1. Сбор и анализ данных от команд. Результат: отчёт, графики
    - еженедельные обновления    
    - балансирующая метрика: не более 2 часов не подготовку отчёта 

2. Определение и анализ критической цепи, визуализация состояния
    - анализ буферов, отклонений, формирование прогнозов и рекомендаций
    - балансирующая метрика: не более 10 минут на презентацию текущего состояния проекта

3. Автоматизация сбора данных от команд, автоматизация создания отчётов
    - балансирующая метрика: не более 10 минут чистого времени в неделю от команды на подготовку и предоставление всех необходимых данных

4. Формирование функции выигрыша для всего проекта
    - стимулировать соблюдение сроков
    - стимулировать развитие проекта в соответствии с ТЗ
    - стимулировать вовлечение каждого члена команды
    - стимулировать взаимопомощь между командами
    - стимулировать генерацию и внедрение идей по улучшению своего проекта, концепции и курса
    - стимулировать минимизацию доверенной вычислительной базы, применение принципов и шаблонов проектирования СКИБ
    - балансирующие метрики: 
        - нулевые финансовые затраты
        - рост удовлетворённости команд месяц-к-месяцу
        - персональные данные не используются        

5. Выработка рекомендаций по улучшению учебных материалов курса
    - использовать материалы из проектов команд (с их согласия)
    - предложить список новых тем и примеров
    - предложить сами занятия на основе запросов команд
    - вовлечь представителей команд в подготовку и проведение семинаров, максимально полезных для других участников
    - балансирующие метрики:
        - лидеры команд одобряют этот движ
        - удовлетворённость команд растёт месяц-к-месяцу

## Что получают команды

знания и навыки
- командной работы, в т.ч. гибкой разработки
- управления проектами
- применения принципов и шаблонов проектирования систем с конструктивной информационной безопасностью
- создания масштабируемых защищённых систем
- применения актуальных инструментов разработки и развёртывания, технологий контейнеризации
- автоматизации тестирования ПО, включая тестирование информационной безопасности
- управления рисками в проекте
- применения современных технологий взаимодействия программных систем, включая технологии передачи критичных данных через недоверенные сети, децентрализованное взаимодействие и распределённые реестры

Кроме того
- успешные команды могут получить приглашение на инженерные соревнования по кибериммунной автономности без квалификационного отбора
- лучшие команды могут получить рекомендательные письма для будущих работодателей
